## 结论摘要

- 结构语义：前端数据结构整体遵循 `sections[] -> chapters[] -> episodes[]` 三层语义，但“新建 section/chapter”链路存在字段缺失导致结构失效（P0）。
- 必填字段：新建节点时存在“未一次性赋齐后端/业务必需字段”的情况（P0），直接导致后端 `chapterId` 为空、前端编号重算崩溃。
- id 策略：sectionId/chapterId 由客户端生成，历史实现依赖 `Date.now()`，存在并发碰撞风险（P1）；episodeId 由后端生成并回填（符合推荐方向）。
- 接口顺序：创建 episode 与更新 outline 结构在代码中存在分叉/缺失，导致“episode 已创建但 outline 未持久化引用”的一致性风险（P0）。
- 时间字段：outline 的 sections JSON 在客户端携带 `createdAt/updatedAt` 并入库（P1）；episode 创建 payload 未携带（符合约束）。

## 针对“典型问题”的根因与修复要点

### 1) 新建 chapter 报错：Cannot read properties of undefined (reading 'length')

- 根因：新建 section/chapter 的对象缺少 `chapters` 数组（或 `episodes` 数组），`recalculateNumbers` 直接访问 `section.chapters.length` 导致崩溃。  
  - 典型触发路径：[ScriptOutline.handleNodeSave](file:///d:/Code/Java/auto-ai-movie/frontend/src/pages/scripts/component/scripts-outline-chapter/ScriptOutline.tsx#L124-L210)
- 修复要点：
  - 新建节点时强制初始化 `chapters: [] / episodes: []` 和计数字段；
  - `recalculateNumbers` 对缺失数组做兜底，保证历史脏数据不会再把页面打崩。

### 2) 新建 episode 后端报错：章节ID不能为空

- 根因：chapter 节点在前端创建时未生成 `chapterId`，后续在 chapter 下创建 episode 时传入的 `chapterId` 为空。  
  - 触发点：[ScriptEpisodeController.createEpisode](file:///d:/Code/Java/auto-ai-movie/ai-movie-scripts/src/main/java/com/liangshou/movie/scripts/adapter/controller/ScriptEpisodeController.java#L33-L70)
- 修复要点：
  - 统一 chapterId 生成与持久化；
  - 在前端调用 `createEpisode` 前做 `chapterId` 非空校验；为空则阻止提交并提示定位字段。

## 与“section → chapter → episode”语义及数据模型一致性核对

- 三层结构语义：前端节点类型与后端 DTO（StoryOutlineDTO/OutlineSectionDTO/OutlineChapterDTO/OutlineEpisodeDTO）字段基本一致。  
- chapter 无独立表：符合后端设计（chapter 仅存在 outline JSON，episode 表以 `chapterId` 字段关联）。  
  - [ScriptsDesign.md](file:///d:/Code/Java/auto-ai-movie/ai-movie-scripts/docs/ScriptsDesign.md#L8-L15)
- 风险点：outline JSON 内的 episode 引用与 episode 表内容是“弱一致”，缺少后端事务聚合接口。

## 回归测试用例（建议）

1. 新建 section：保存后 sections 数量 +1；sectionId 非空；chapterCount=0；页面不报错。
2. 新建 chapter：在指定 section 下保存后 chapters 数量 +1；chapterId 非空；episodeCount=0；页面不报错。
3. 新建 episode：在指定 chapter 下保存后 episodes 数量 +1；episodeId 非空；后端 `scripts_episodes` 中存在对应记录；outline JSON 中引用存在。
4. 幂等性（重复点击保存）：不会产生重复 episode 引用；后端若发生唯一键冲突应返回 409，前端提示定位。
5. 弱网/超时：保存出现超时时，展示“保存中…”并自动重试 3 次；仍失败则给出重试按钮。
